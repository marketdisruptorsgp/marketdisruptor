# Fix: Redesign Routing + Add User Workflow Section to Intel Report

## Diagnosis

### Issue 1: Redesign Routing

The route `/analysis/:id/redesign` exists in `App.tsx` (line 57) and the `RedesignPage.tsx` component is correctly implemented. However, the **"Continue where you left off" banner** calls `handleLoadSaved`, which navigates to either:

- `/analysis/${id}/report` for product analyses (line 471 of AnalysisContext.tsx)
- `/analysis/${id}/disrupt` for first_principles analyses (line 460)

It never navigates to the redesign step. More critically, the **StepNavigator inside `Index.tsx**` (the fallback/old inline flow at line 721-736) uses `setActiveStep(s)` for inline step switching, NOT route-based navigation. When users are on `Index.tsx` (the `/` route), clicking "Redesign" in the step navigator switches the inline view correctly -- but if there is a mix-up between the two flows (inline vs routed), navigation breaks.

The actual routing between dedicated pages (DisruptPage -> RedesignPage -> StressTestPage) is correct in the code. The likely culprit is either:

1. The user is in the Index.tsx inline flow and clicking "Next Step" buttons that don't exist (no `NextStepButton` between steps in Index.tsx inline flow).
2. Or, state is lost on page refresh because `analysis.step` resets to "idle" (since AnalysisContext uses `useState`, not persisted state), causing the guard `analysis.step !== "done"` to redirect to `/`.

**Root cause confirmed**: If a user refreshes the browser on `/analysis/:id/redesign`, the AnalysisContext state resets to `step: "idle"`, `selectedProduct: null`. The guard on line 16 of RedesignPage immediately redirects to `/`. This affects ALL step pages equally, but users report it on Redesign because it's the newest addition.

**Fix**: The routing code itself is correct. The fix needs to handle state restoration when navigating directly to step URLs. For now, we should ensure the inline flow in Index.tsx properly has "Next Step" buttons between all steps, and the routed flow works when navigated from the previous step.

### Issue 2: User Workflow Section Missing from Intel Report

The `ReportPage.tsx` DETAIL_TABS (line 53-59) only includes: Overview, Community Intel, Pricing Intel, Supply Chain, and Patent Intel. There is NO "User Workflow" or "User Journey" tab, even though:

- The `WorkflowTimeline` component exists and is exported from `FirstPrinciplesAnalysis.tsx`
- The `INTEL_SECTION_DESCRIPTIONS` includes a `workflow` entry
- The `SECTION_DESCRIPTIONS` in `stepConfigs.ts` includes `workflow: "Step-by-step journey & friction points"`
- The user workflow data (`userWorkflow`) is part of the `FirstPrinciplesData` interface, which comes from the disrupt analysis, NOT the initial report data

The User Workflow data is generated by the `first-principles-analysis` edge function (disrupt step), so it only becomes available AFTER the user runs the Disrupt analysis. It should be shown in the Intel Report as an enrichment section that appears once disrupt data is available, or as a section within the Disrupt step itself.

**Fix**: Add a "User Workflow" tab to the ReportPage DETAIL_TABS that renders the `WorkflowTimeline` component using disrupt data when available.
